# CLI Agent Launchers

## Role

This subsystem exposes external coding agents as first-class runtime tools.

Instead of treating Claude Code, Codex, Gemini, OpenCode, Cursor Agent, or
Copilot as separate user-managed CLIs, the desktop app installs wrappers and
hooks so they participate in the product's workspace and notification model.

## Why it matters

One of the system's distinctive capabilities is that agent CLIs can be opened
and run directly from the product environment while still reporting lifecycle
events back into the UI.

This is the bridge between raw terminal agents and the higher-level
superbuilder workflow.

## Supported launch surfaces

The most visible agent command surface is defined in:

- `packages/shared/src/agent-command.ts`

That shared package currently models these agent types:

- `claude`
- `codex`
- `gemini`
- `opencode`
- `copilot`
- `cursor-agent`

The desktop runtime also contains wrapper support for additional tools such as
Factory Droid and Mastra-oriented hooks.

## Libraries and services

- shell wrapper scripts generated by desktop
- per-agent hook configuration files
- desktop notification server
- Claude Code SDK usage in feature-stack execution flows
- Gemini, OpenAI, and Claude model/provider support in feature-stack AI services

## Runtime shape

There are two related but different paths here.

### Desktop CLI wrappers

Desktop creates workspace-scoped wrapper binaries under the Superset-managed
home directory and adds them to shell PATH. Those wrappers inject hook
configuration, notification scripts, and environment-specific config locations.

### Feature-server agent execution

The feature-stack also has a server-side execution path in `agent-desk`. That
path creates worktrees, runs Claude Code programmatically, streams execution
events, then performs validation and Git/PR follow-up.

This means the repository supports both:

- user-visible local CLI agent launching
- backend-mediated agent execution for structured feature workflows

## Event flow

1. desktop writes wrappers and hook files into workspace-scoped directories
2. shell integration exposes those wrappers as the visible agent commands
3. when an agent runs, hook scripts report start, stop, permission, and tool
   events back to desktop
4. desktop updates tab and pane state using notification subscriptions
5. in structured server-driven flows, `agent-desk` executes a similar pattern
   from the backend side rather than from the local shell

## Constraints and migration notes

- some agents use workspace-local configuration, while others still require
  merged global settings files
- the CLI surface is broader than the model-provider surface; Gemini exists both
  as a CLI wrapper path and as an API model provider path
- server-side execution is currently most tightly coupled to Claude Code, even
  though the product exposes multiple local agent CLIs
