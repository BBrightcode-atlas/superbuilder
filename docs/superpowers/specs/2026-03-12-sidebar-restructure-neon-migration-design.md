# Sidebar Restructure + Supabase → Neon Migration

## Overview

Reorganize the Desktop atlas sidebar into two sections (Features / Builders) and replace Supabase with Neon MCP for database project creation in the Composer pipeline.

## 1. Sidebar Restructure

### Current
Single "Features" header with 4 flat nav items: Catalog, Studio, Composer, Deployments.

### Target
```
┌─────────────────┐
│ Features        │  ← Section header
│   Catalog       │
│   Studio        │
├─────────────────┤
│ Builders        │  ← New section header
│   Composer      │
│   Deployments   │
└─────────────────┘
```

### Implementation
- `AtlasSidebar.tsx`: Convert flat `NAV_ITEMS` array to grouped sections
- Route paths (`/atlas/catalog`, `/atlas/studio`, `/atlas/composer`, `/atlas/deployments`) remain unchanged

## 2. Neon MCP Integration (replacing Supabase)

### Token Management
- Env variable: `NEON_API_KEY`
- DB fallback: `atlasIntegrations.service = "neon"` (encrypted storage)
- UI: Token input form shown when not configured (same pattern as current Supabase)

### New tRPC Router: `neon.ts`

| Procedure | Description | Neon MCP Tool |
|-----------|-------------|---------------|
| `saveToken` | Validate + encrypt store Neon API key | N/A (direct API validation) |
| `removeToken` | Delete stored token | N/A |
| `getConnectionStatus` | Check env or DB for token | N/A |
| `listOrganizations` | List Neon organizations | `list_organizations` |
| `createProject` | Create Neon project + save to `atlasProjects` | `create_project` |
| `getConnectionString` | Get project connection string | `get_connection_string` |
| `writeEnvFile` | Write `DATABASE_URL` + `NEON_PROJECT_ID` to .env | N/A |

### .env Output
```
# Neon (auto-generated by Composer)
DATABASE_URL=postgres://...
NEON_PROJECT_ID=...
```

## 3. Composer Pipeline Changes

### Step Labels
```
Before: Feature 선택 → 의존성 확인 → 프로젝트 설정 → 프로젝트 생성 → Supabase → Vercel
After:  Feature 선택 → 의존성 확인 → 프로젝트 설정 → 프로젝트 생성 → Neon     → Vercel
```

### Key Differences from Supabase
- No health check polling — Neon projects are available immediately after creation
- Connection string based — no separate anon/service keys
- Neon MCP handles project lifecycle

### UI: `NeonSetup` component (replacing `SupabaseSetup`)
- Token input + validation
- Organization selection dropdown
- Region selection (default: `aws-ap-northeast-2`)
- "Create" and "Skip" actions

## 4. Deployments Changes

### Project Deletion
Replace Supabase project deletion with Neon MCP `delete_project` call.

### Display
Show Neon project info instead of Supabase URL in `DeploymentCard`.

## 5. Local DB Schema Changes

`atlasProjects` table column changes:
- `supabaseProjectId` → `neonProjectId`
- `supabaseProjectUrl` → `neonConnectionString`
